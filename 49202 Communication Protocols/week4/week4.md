## 용어 정리

## iperf3란?

**네트워크 성능 측정 도구**야.

두 컴퓨터 사이에서 데이터를 실제로 전송해보고, 그 결과로 **처리량(Throughput), 지연시간(Latency), 패킷 손실(Packet Loss), 지터(Jitter)** 같은 수치를 측정해줘.

한 쪽은 **서버 모드**(`-s`)로 대기하고, 다른 쪽은 **클라이언트 모드**(`-c`)로 접속해서 데이터를 쏘는 방식이야. 실제 인터넷에서 "내 네트워크가 얼마나 빠른지" 측정할 때 엔지니어들이 쓰는 도구야.

오늘 실습에서는:
- h4 = 서버 (받는 쪽)
- h1 = 클라이언트 (보내는 쪽)

---

## OSPF란?

**Open Shortest Path First** — 라우터들이 서로 경로를 자동으로 찾아주는 **라우팅 프로토콜**이야.

우리 실습 네트워크엔 r1~r9까지 9개의 라우터가 있어. h1에서 h4로 데이터를 보낼 때, 그 데이터가 어떤 라우터를 거쳐서 가야 하는지를 자동으로 계산해주는 게 OSPF야.

OSPF는 **링크 상태(Link State)** 방식을 써서, 각 라우터가 네트워크 전체 지도를 가지고 있고 **가장 짧은 경로(Shortest Path)**를 계산해. 그래서 `start.py` 실행 후 30초를 기다린 거야 — 9개 라우터들이 서로 대화하면서 경로 정보를 교환하는 시간(수렴, Convergence)이 필요하거든.

---

## ping이란?

**두 호스트 사이의 연결 상태와 지연시간을 확인하는 도구**야.

작동 원리는 단순해:
1. h1이 h5한테 "야 거기 있어?" (ICMP Echo Request) 패킷을 보냄
2. h5가 "응 나 여기 있어" (ICMP Echo Reply) 패킷을 돌려보냄
3. h1이 그 왕복 시간을 측정해서 출력해줌 → 이게 **RTT (Round Trip Time)**

오늘 실습에서 ping을 h5로 계속 보낸 이유는, iperf3가 TCP/UDP로 대량의 데이터를 전송할 때 **그 트래픽이 ping의 응답시간에 영향을 주는지 관찰**하기 위해서야. 네트워크가 혼잡해지면 ping의 응답시간도 늘어나거든.

---

## TCP Reno, TCP-BBR, TCP-Cubic이란?

셋 다 **TCP 혼잡 제어 알고리즘(Congestion Control Algorithm)**이야.

TCP는 데이터를 보낼 때 네트워크가 얼마나 혼잡한지 파악하면서 전송 속도를 조절해야 해. 그 조절 방식이 알고리즘마다 달라.

---

### TCP Reno
가장 **고전적인** TCP 혼잡 제어 방식. 강의에서 배운 그것.

- **슬로우 스타트(Slow Start)**: 처음엔 작은 윈도우로 시작해서 빠르게 증가
- **혼잡 회피(Congestion Avoidance)**: 임계값(ssthresh) 이후엔 천천히 선형 증가
- **패킷 손실 감지** 시 전송 속도를 절반으로 확 줄임
- **문제점**: 패킷 손실이 발생해야만 혼잡을 감지함 → 이미 늦은 거야

---

### TCP Cubic
현재 **Linux의 기본 TCP 알고리즘** (Reno의 개선판).

- 혼잡 감지 후 윈도우를 줄였다가, **3차 함수(Cubic function)** 형태로 빠르게 회복해
- 고속/장거리 네트워크에서 Reno보다 훨씬 효율적
- 마찬가지로 패킷 손실 기반으로 혼잡을 감지함

---

### TCP BBR
Google이 2016년에 개발한 **완전히 다른 방식**의 알고리즘.

- Reno/Cubic과 달리 **패킷 손실 기반이 아님**
- 대신 **대역폭(Bandwidth)과 RTT(지연시간)**을 직접 측정해서 네트워크 상태를 모델링함
- "지금 네트워크가 얼마나 빠른지"를 능동적으로 파악하면서 최적 속도 유지
- 혼잡이 생기기 **전에** 속도를 조절하므로 이론적으로 더 효율적

---

## h1에서 h4로 30초 동안 뭘 보낸 건지?

실제로 보낸 내용은 **아무 의미 없는 랜덤 데이터(pseudorandom data)**야.

iperf3는 실제 파일이나 웹페이지 같은 걸 보내는 게 아니라, **네트워크 성능 측정이 목적**이기 때문에 그냥 의미없는 데이터를 최대한 빠르게 채워서 보내.

오늘 실행한 명령어를 분해하면:

```
iperf3 -c h4 -t 30 -O 5 -C reno -Z
```

| 옵션 | 의미 |
|------|------|
| `-c h4` | h4에 클라이언트로 접속 |
| `-t 30` | 30초 동안 전송 |
| `-O 5` | 처음 5초는 통계에서 제외 (슬로우 스타트 구간이라 측정값이 튀기 때문) |
| `-C reno` | TCP Reno 혼잡 제어 알고리즘 사용 |
| `-Z` | Zero-copy 방식으로 전송 (CPU 부하 줄이기 위해) |

결과적으로 **30초 동안 약 342 MB의 랜덤 데이터를 평균 95.7 Mb/s 속도로 h4에 전송**한 거야. 그리고 그 전송 과정에서 TCP Reno가 어떻게 속도를 조절하는지를 Wireshark 그래프로 시각화해서 확인한 거고.

---

## 절대적 기준이 없는 이유

95.7 Mb/s가 좋은지 나쁜지는 **링크 속도가 뭔지 모르면 판단 불가**야. 링크가 1 Gb/s짜리면 10% 밖에 못 쓴 거니까 나쁜 거고, 링크가 100 Mb/s짜리면 거의 다 쓴 거니까 좋은 거야.

---

## 그럼 어떻게 판단하냐?

### 1. 링크 이론값 대비 비교
오늘 실습 네트워크(Mininet)의 링크는 기본 설정이 **100 Mb/s**야. 그런데 결과가 95.7 Mb/s 나왔으니까 **이론 최대값의 약 95%** → 꽤 효율적인 거야.

### 2. 오버헤드 감안
TCP + IP + Ethernet 헤더 오버헤드가 있으니까 100 Mb/s 링크에서 실제 페이로드가 100 Mb/s 나오는 건 불가능해. 95% 나왔으면 실제로는 매우 좋은 결과야.

### 3. Retr(재전송) 횟수
오늘 결과에서 **Retr = 22**가 나왔는데, 이건 30초 동안 22번 패킷 손실이 있었다는 뜻이야. 많냐 적냐는 상대적이지만, 전체 전송 패킷 수 대비 22번은 **매우 적은 편**이야.

### 4. 알고리즘끼리 비교
이게 오늘 실습의 핵심이야. TCP Reno, Cubic, BBR을 **같은 조건에서 돌려서 서로 비교**하는 거야:
- 누가 더 높은 Throughput을 냈나
- 누가 Retr이 더 적었나
- Wireshark 그래프에서 누가 더 안정적으로 속도를 유지했나

---

## 오늘 결과 요약 평가

| 항목 | 결과 | 평가 |
|------|------|------|
| Throughput | 95.7 Mb/s | 100 Mb/s 링크에서 95% → 양호 |
| 재전송(Retr) | 22회 | 적은 편 → 양호 |
| Cwnd 증가 | 1.08 → 4.12 MBytes | 꾸준히 증가 → 정상 동작 |

지금 이 결과는 **기준값(Baseline)** 역할을 하는 거야. 이다음에 TCP Cubic, BBR로 같은 실험을 하고 이 숫자들과 비교하면 어떤 알고리즘이 더 효율적인지 판단할 수 있어.

## 링크(Link)란?

두 장치를 연결하는 **물리적/논리적 통신 선로**야.

예를 들어 h1과 r1 사이의 연결선 하나가 링크야. 이 링크는 **최대 전송 속도(대역폭, Bandwidth)**가 정해져 있어. 오늘 Mininet 실습 네트워크는 링크당 기본 **100 Mb/s**로 설정되어 있어. 즉, 이 선로를 통해 초당 최대 100 Mb의 데이터만 흐를 수 있어.

현실에서는 집에 있는 랜선, Wi-Fi 신호, 광케이블 등이 다 링크야.

---

## Throughput이 높다는 게 링크의 몇%를 차지했다는 뜻이 맞는건지?

맞아. 정확히 그 뜻이야.

**Throughput(처리량)** = 실제로 단위 시간당 전달된 데이터 양

링크 속도가 **최대로 가능한 양**이라면, Throughput은 **실제로 전달된 양**이야.

오늘 결과:
- 링크 최대 속도: 100 Mb/s
- 실제 Throughput: 95.7 Mb/s
- 효율: 95.7 / 100 = **95.7%**

수도관에 비유하면, 링크는 수도관의 굵기(최대 물 흐름), Throughput은 실제로 흐른 물의 양이야.

---

## Throughput % 가 높으면 좋은건지?

**대부분의 경우 높을수록 좋아. 단, 상황에 따라 달라.**

높은 게 좋은 경우:
- 파일 전송, 동영상 스트리밍 → 빠를수록 좋으니까 링크를 최대한 활용하는 게 좋음

높다고 무조건 좋지 않은 경우:
- 링크를 100% 가까이 쓰면 **다른 트래픽이 끼어들 공간이 없어져서** 혼잡(Congestion)이 발생함
- 그래서 TCP 혼잡 제어 알고리즘이 존재하는 거야 — 링크를 100% 쓰되, 혼잡이 생기지 않는 선에서 최대한 쓰는 게 목표

---

## Retr=22, 전체 패킷 수는 어떻게 아나?

Wireshark 하단에 이렇게 나왔었어:

```
Packets: 342,269
```

즉 30초 동안 캡처된 전체 패킷 수가 **342,269개**야.

그 중에서 재전송된 게 22번이니까:

22 / 342,269 = **약 0.006%**

이건 극히 적은 수치야. 현실 네트워크에서 1% 미만이면 양호한 수준으로 봐. 다만 오늘은 Mininet 가상 환경이라 실제 물리 네트워크보다 훨씬 깨끗한 환경이기 때문에 재전송이 적게 나온 것도 있어.

---

## Cwnd(Congestion Window)란?

**TCP 송신자가 ACK(수신 확인)를 받기 전에 한꺼번에 보낼 수 있는 최대 데이터 양**이야.

쉽게 비유하면:

> 택배를 보낼 때, 상대방이 "받았어"라고 연락 오기 전에 내가 먼저 보낼 수 있는 택배 박스 수

- Cwnd가 크면 → 한번에 많이 보낼 수 있음 → 빠름
- Cwnd가 작으면 → 조금씩 보내고 확인 받아야 함 → 느림

오늘 결과에서 Cwnd가 이렇게 변했어:

| 시간 | Cwnd |
|------|------|
| 0~1초 | 1.08 MBytes |
| 5~6초 | 2.20 MBytes |
| 15~16초 | 2.71 MBytes |
| 29~30초 | 4.12 MBytes |

처음에 작다가 점점 커지는 게 보이지? 이게 바로 TCP Reno의 동작 방식이야:

1. **슬로우 스타트**: Cwnd를 처음엔 작게 시작해서 빠르게 2배씩 증가
2. **혼잡 회피**: 임계값에 도달하면 천천히 조금씩 증가
3. **패킷 손실 발생**: Cwnd를 확 줄임 (Retr 22번이 발생한 순간들)
4. **다시 증가 반복**

Wireshark 그래프에서 초반에 급격히 올라가다가 이후 완만하게 올라가는 게 바로 1→2→3→4 과정을 시각화한 거야.

두 곳에서 각각 봤어.

---

**Retr = 22** → iperf3 터미널 출력 마지막 줄

아까 h1 창 하단에 이렇게 나왔었어:
```
[ 7] 0.00-30.00 sec  342 MBytes  95.7 Mbits/sec  22   sender
```
여기 `22`가 Retr 값이야.

---

**342,269개** → Wireshark 하단 상태바

아까 스크린샷 중에 Wireshark 하단에 이렇게 표시됐었어:
```
Packets: 342269 · Displayed: 342031 (99.9%) · Dropped: 0 (0.0%)
```
필터 적용 전 전체 캡처 패킷 수야.

---

다만 한 가지 정정할게. 아까 내가 **"342,269개 중 22번 재전송"** 이라고 했는데 이건 엄밀히 따지면 정확하지 않아.

342,269는 **Wireshark가 캡처한 전체 패킷** (ping ICMP, OSPF, TCP 전부 포함)이고, Retr 22는 **iperf3 TCP 스트림에서만의 재전송**이야. 정확하게 비교하려면 TCP 패킷만 따로 세야 해. 비율 자체는 여전히 매우 낮지만, 분모를 잘못 쓴 거야.

### 질문 : Wireshark가 캡처한 전체 패킷 (ping ICMP, OSPF, TCP 전부 포함)이고, Retr 22는 iperf3 TCP 스트림에서만의 재전송 -> 왜 이런 차이가 발생하는거야. ping ICMP는 뭐고 OSPF는 뭐야?

## 왜 차이가 발생하냐?

Wireshark는 **h1의 네트워크 인터페이스(h1-eth0)를 지나가는 모든 패킷**을 캡처해. iperf3 데이터든, ping이든, 라우터끼리 주고받는 신호든 **종류 구분 없이 전부 다 잡아.**

반면 iperf3는 **자기가 보낸 TCP 데이터 패킷만** 추적하고 있어.

그래서 Wireshark 총 패킷 수 안에는 여러 종류의 패킷이 섞여 있는 거야:

---

## ping ICMP란?

**ICMP(Internet Control Message Protocol)** 는 TCP나 UDP처럼 데이터를 전송하는 프로토콜이 아니라, **네트워크 상태를 확인하기 위한 제어용 메시지 프로토콜**이야.

ping 명령어가 바로 이 ICMP를 사용해. h1에서 `ping h5`를 실행하면:

1. h1이 h5한테 **ICMP Echo Request** 패킷을 보냄
2. h5가 **ICMP Echo Reply** 패킷으로 응답
3. 이 왕복이 1초에 한 번씩 계속 발생

우리가 ping을 30초 동안 실행했으니까 약 **60개의 ICMP 패킷**(Request 30 + Reply 30)이 Wireshark에 잡혔어. TCP 스택 위에서 동작하는 게 아니라 완전히 별개의 프로토콜이기 때문에 iperf3는 이걸 전혀 모르고, Wireshark만 다 보고 있는 거야.

---

## OSPF 패킷이란?

앞서 OSPF가 라우터들이 경로를 자동으로 찾는 프로토콜이라고 했잖아. 그 과정에서 라우터들은 **지속적으로 서로에게 메시지를 주고받아.**

대표적으로 **Hello 패킷**이 있어. 각 라우터가 "나 아직 살아있어, 나랑 연결된 링크 정보는 이거야"라고 주변 라우터들에게 **주기적으로 브로드캐스트**하는 메시지야. 기본적으로 **10초마다** 한 번씩 보내.

Wireshark 스크린샷에서 아까 노란색으로 표시된 패킷들이 바로 이 OSPF Hello 패킷이었어:
```
224.0.0.5   OSPF   78   Hello Packet
```
h1도 r1과 연결되어 있으니까 r1이 보내는 OSPF 패킷이 h1의 인터페이스에도 보여서 Wireshark가 잡은 거야.

---

## 정리하면

Wireshark에 찍힌 342,269개 패킷의 구성은 대략 이렇게 돼:

| 패킷 종류 | 발생 원인 | 비율 |
|-----------|-----------|------|
| TCP (iperf3 데이터) | h1→h4 데이터 전송 | 대부분 (99% 이상) |
| ICMP (ping) | h1→h5 ping | 극소량 |
| OSPF | 라우터들의 경로 교환 | 극소량 |

iperf3가 30초 동안 100 Mb/s 가까이 데이터를 쏴댔으니까 TCP 패킷이 압도적으로 많고, ICMP랑 OSPF는 사실상 무시할 수준이야. 그래서 Wireshark 필터에서 `tcp.stream eq 1`을 적용하면 iperf3 TCP 패킷만 걸러지고 나머지는 사라지는 거고.
